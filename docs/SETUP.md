# Setup Guide (one-time)

This guide configures a new project created from the template. **You only need it once** — after running setup you can delete this file (and the setup tooling itself, see below).

---

## 1. Create the project

1. On GitHub, click **Use this template → Create a new repository**.
2. Clone it locally and open it.

## 2. Run the configurator

```bash
pnpm setup-project
```

It asks for:

- **Project name** and **package scope** (default `@repo`, e.g. `@acme`).
- **Auth provider** — **Custom API (default)**, Supabase, Firebase, or None/Mock.
- Provider config inline:
  - Custom API → **base URL** (leave blank for demo mode).
  - Supabase → URL + anon key. Firebase → API key + auth domain + project ID.
- Whether to include the **Admin Panel**, **Web Auth Pages**, **Users Module**, **Dashboard**.
- Whether to **remove the setup tooling** when done (recommended — it's one-time).

Based on your answers it automatically:

- Writes **`apps/main/.env.local`** with your values.
- Regenerates `apps/main/src/config/auth.ts` and `apps/main/next.config.js` (correct `transpilePackages`).
- Deletes unused auth adapters / example modules and their dependency entries.
- Applies your package scope across the repo.
- If you opted in, deletes `scripts/setup-project.ts`, the `setup-project` npm script, and `ts-node`.

## 3. Install & run

```bash
pnpm install   # relinks workspace packages after pruning
pnpm dev
```

- Site → http://localhost:3000
- Admin → http://localhost:3000/admin

## Environment variables

> **Critical:** env files live in **`apps/main/`**, not the repo root — Next.js only loads `.env*` from the app directory. Copy `apps/main/.env.example` → `apps/main/.env.local` (or let `setup-project` write it). Only `NEXT_PUBLIC_*` vars reach the browser. **Restart `pnpm dev` after changing env values** (they are inlined at server start).

| Variable                    | Purpose                                              |
| --------------------------- | ---------------------------------------------------- |
| `NEXT_PUBLIC_API_BASE_URL`  | Base URL for auth + all data requests (admin + web). |
| `NEXT_PUBLIC_AUTH_PROVIDER` | Informational: which provider is active.             |
| `NEXT_PUBLIC_SUPABASE_*`    | Supabase config (if using Supabase).                 |
| `NEXT_PUBLIC_FIREBASE_*`    | Firebase config (if using Firebase).                 |

## Smart fallback (Custom API default)

The default Custom API setup uses one switch driven by `NEXT_PUBLIC_API_BASE_URL` (see `apps/main/src/config/env.ts`):

- **Empty** → mock auth + mock data. A fresh clone runs end-to-end with no backend; sign in with any credentials.
- **Set** → the whole app (auth + every data service) uses the real API automatically.

To go live, set the base URL and restart — **no code changes**:

```bash
# apps/main/.env.local
NEXT_PUBLIC_API_BASE_URL=https://api.yourbackend.com
```

The mock branch is dead-code-eliminated in production builds when a URL is set, so it costs nothing in a real deployment.

## The config layer (single source of truth)

`apps/main/src/config/`:

| File      | Responsibility                                                                                         |
| --------- | ------------------------------------------------------------------------------------------------------ |
| `env.ts`  | Reads `NEXT_PUBLIC_API_BASE_URL` once; exposes `IS_API_CONFIGURED`.                                    |
| `api.ts`  | The **single shared `apiClient`** (base URL + bearer token from the auth store) — used by admin + web. |
| `auth.ts` | Selects the active auth adapter. Regenerated by `setup-project`.                                       |

## Auth provider contract (Custom API)

`packages/auth-custom-api/src/adapter.ts`. Default endpoints (relative to the base URL):

| Action       | Request                        |
| ------------ | ------------------------------ |
| signIn       | `POST /admin/v1/auth/login`    |
| signUp       | `POST /admin/v1/auth/register` |
| signOut      | `POST /admin/v1/auth/logout`   |
| current user | `GET /admin/v1/auth/me`        |

Login must return `{ access_token, user }` — an outer `{ success, code, data }` envelope is unwrapped automatically. `user.is_admin: true` grants admin access. Override paths via the adapter's `endpoints` option in `config/auth.ts`. On every load the app revalidates the session via `/me`; a `401` signs the user out.

### Adding a new auth adapter

1. Create `packages/auth-<name>`.
2. Implement the `AuthAdapter` interface from `@repo/auth-core` (`signIn`, `signUp`, `signOut`, `getSession`, `getCurrentUser`, `refreshSession`, `forgotPassword`, `resetPassword`). Adapters are **stateless** — return a `Session`/`User`; the auth store owns persistence.
3. Wire it in `apps/main/src/config/auth.ts`.

## Troubleshooting

| Symptom                               | Cause / fix                                                                |
| ------------------------------------- | -------------------------------------------------------------------------- |
| Login hits `localhost:3000/...` (404) | Env not loaded → ensure `apps/main/.env.local` exists and **restart dev**. |
| "Checking session…" forever           | A workspace package isn't in `transpilePackages` (setup regenerates this). |
| Logged out on refresh                 | `/admin/v1/auth/me` returns `401` for a valid token — check the endpoint.  |
| Type errors after switching providers | Run `pnpm install` to relink workspace packages.                           |
